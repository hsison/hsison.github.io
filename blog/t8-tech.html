Test Driven Development aka TDD

The brain,Kent Beck, is the one who rediscovered and developed this technique into what it is today.
Test-driven development is a software development process that relies on the repetition of a very short development cycle. First, the developer writes an automated test case that defines a desired improvement or new function, and then produces the minimum amount of code to pass that test. Programmers also apply the concept to improve and debug code more efficiently. You might say,"This TDD stuff sounds amazing! How can I use this?"
You're in luck. Let's do this.

How To Use TDD

Write A Test

The first part to TDD is writing the test. Yes, you write the test before you write the code don't freak out. To write a test you must have a clear understanding of what you want the code to do, i.e. specifications and requirements. You can use cases and user stories to do this.

Run the test

You do this to make sure the new test doesn't mistakenly pass without requiring new code and also to make sure the new feature doesn't already exist in your program. The new test should also fail for the expected reason.

Get to writin' code

Write some code that will cause the test to pass. This bit of code will probably be ugly but that's why we refactor later. For now, just make sure the code makes the test pass and you can beautify it later.

Run the test again

The tests should pass now. If they don't, rewrite your code.

Refactor that ugly code

Once your tests and code are happy it's time to beautify. Discover some better variable names, method names, class names, object names, etc. If there's any repetition make sure to clean that up too.
