Classes vs Modules

Before you dive into this super cool blog post about Classes and Modules you should already have a pretty good understanding of what classes are and how they operate. If you don't, no worries! I gotcha covered. Check out my previous post Sass, A$%, and Classes and then come on back.

What in the world are modules?

No, no, it's not a disease.
The first thing I always do when I'm unsure about anything in life is Google it. I'll give you a moment to do so. (....insert elevator music here....)
If you've already continued reading because you're too lazy to look it up here you go.
According to Google a Module is a set of standardized parts or independent units that can be used to construct a more complex structure, such as an item of furniture or a building.
For all of the manly men out there, you can think of module as all of the parts you need to build a car, i.e., an engine, brakes, tires, etc. For my ladies, you can think of it as a makeup. Moving on.

What do modules have to do with classes?

Modules are very much like classes in the way they're written which makes it pretty easy to understand if you already have a good grip on classes. The difference between the two is that modules can't create instances and can't have subclasses. They're simply used to simply store things.
Check out this sweet example:

module MyLibrary
FAVE_BOOK = "Book about Hannah"
end
Did you notice that the module syntax is really close to class syntax?
Just like class names, the module names are written using Capitalized CamelCase.
Another difference is modules don't contain variables. Because variables are subject to change we don't want to include them in our modules. In modules we use constants(because they typically stay the same)! Ruby constants are written using all caps and are separated with underscores if there are multiple words like this:
Ex: HANNAH_ROCKS

Let's get weird with it!

One of the main purpose of modules in Ruby is to separate methods and constants into a specifically named space, which is actually called 'namespacing'. Real clever right?
Namespacing is a way of bundling logically related objects together. It allows classes or modules with conflicting names to co-exist while avoiding collisions*BAM*.
No I'm not rambling here's the point.
Modules can actually hold classes. Hold on let me just say that again, Modules can actually hold classes.Instead of me trying to explain what that means to you, I'll show you.

Example: module Hannah
class Rocks
def initialize
@cool = 100
end
end
end
cool_rocks = Hannah :: Rocks.new
lame_rocks = Rocks.new
p cool_rocks.class
p lame_rocks.class

Did you see that? It doesn't affect our class at all! We have two classes parallel to one another because we've namespaced the Rocks class under the Hannah module. That weird double colon thingy '::' is a constant lookup operator that ONLY looks up Rock in the Hannah module. I told you we were going to get weird. The only reason we're allowed to do all this cool jazz in Ruby is because we namespaced.
